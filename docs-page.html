Introduction Last updated: 2023-14-01

v0dka Instance Launcher is a project I have written in Python and is targeted towards Linux-based systems. I saw an issue with existing products out there, both free and paid solutions, not offering Linux support or if they did have Linux support, they lacked a lot of features that I was looking for.

The closest match I have found is a system called OpenWorldServers, developed by a great guy called Dartanlla. ( Click Click here for their website. ) This is a really good project, however, there is no official Linux support for their system. OWS does work for Linux, but any Linux-related issues you are left to solve yourself or rely on the community to help resolve.

The idea of this system is that you would build your own character persistence API, giving you true freedom of choosing which language to write this in, whether it is C#, Java, PHP, JavaScript, Python... the list goes on, the choice is yours, and your server spin up/shut down/management is all handled through the vodka instance manager.

It should be noted that I will not accept PRs for Windows-related "fixes" as this product does not and will not cover Windows. You are more than welcome to modify the source code to make it work for Windows.

How it works

The system is split into two parts: the API and the Instance Manager.

API:

The API is a web service written in Python using the Flask framework, it connects to a MySQL database for information. The API exposes routes (URLs) which different systems can interact with to perform certain tasks.

Instance Manager:

The Instance Manager makes requests to the API using different routes (URLs). The two main routes for the program are /status and /spin_me_up. The /status route will return a list of tasks that the system should perform such as shutting down instances, performing updates, etc. whereas the /spin_me_up route will return a list of instances waiting to be created. Once the Instance Manager has received a message, it will take the appropriate actions.

Why Linux Not Windows?

As someone who has spent their life working with Linux, I find that it can be difficult to use Windows systems for multiple reasons. The Linux command line is very powerful, whereas Windows has a lot of loopholes to jump through to perform certain tasks. Additionally, the cost of Windows servers is not cheap, and they are licensed per CPU core, while Linux servers are free.

Feature Linux Windows
Cpu/Ram usage(no additional software) depends on distro, but average idle usage can be between 100mb and 1gb and most distro's can run on 1 core requires minimum 2gb of memory and 4 cores
License Cost Free unless you pay for support from RedHat Expensive as you are charged per cpu core, if you have under 16 core's you are charged for a 16core license, if you want to do virtualization this also introduces more costs as with most server licenses its 1 virtual machine
Updates Can update the system in 99.9% of cases without having to reboot(yes even kernal updates) the machine having almost zero downtime Windows updates are a hassle and not being updated can put the server at risk, installing these updates requires server restart which means you have downtime
OS Speed Fast and better at managing memory/process's Slow and does not handle process's or memory as good as linux
Support not great support and is mainly community unless you have a RedHat license Actually very good support you have Microsoft call centres/forums and community to resolve your issues

API

This is all the possible end points you can comminate with and the parameters which they take with a description on what they do

All routes are handled in the main.py file and each route will call a corresponding function in the server.py file this was my attempt at trying to keep code organized however this will no doubt change in the future and docs will be updated to reflect these changes

Lets take a look at an example

This it the Main.py and we are looking at the route create (example.com:8090/create)

## this route is to create instance in database
@app.route("/create")
def create():
args = request.args
server_ip = args['ip']
map_name = args['map']
port = args['port']
return server.create(connection, cursor, server_ip, map_name, port)

We define a wrapper @app.route with the value of /create and assign a function called create to this wrapper

we then fetch the parameters and because this one is handled as a get request we use request.args to get the parameters, further in the code you will see me use request.form for POST requests and request.json for POST json requests anyway i store each value to a variable (you don't need to do this but i find it easier to read) and pass these values to the server.create() function
if you are creating your own functions you will want to pass the connection and cursor values to the function if you are doing any database related stuff

INFO!

A get request is data which is obtained from the url an example of this would be
example.com/create?name=test&lastname=test2
and in our code we could access this

args = request.args
firstname = args['name']
surname = args['lastname']
A Post request sends the data in the body of the request and the only way we can really visualize this is with a inspection tool like telerik inspector or chrome/mozzilla dev tools to analyze the request, i will expand on this in the future with more info

Instance Manager

the instance manager is part of the system which creates our dedicated servers, it talks to our api and gets actions to perform such as start, stop, update, update, reboot etc..
We have 3 scripts (not including config file) to manage the program

Start.py

start.py is our entry point this is how we control everything and this is where we pass our arguments to do certain actions

instance_starter.py

parameters are passed to this script from start.py to start our unreal server instances

instance_manager.py

This is either started manually by running instance_manager.py or if you pass the command start to start.py python3 start.py start then it will start this script, the instance manager talks to the api and then tells start.py what to do with the action received.

Integration

You can implement this in any way you wish and see fit. However, my recommendation would be to create it as a component that you attach to your game mode. Why game mode? Well, in the future, you can tie it into prelogin/postlogin pretty easily. The game mode is typically developed to be run on the server and replicated to clients, so for me, this just makes a logical place to put the code.

The code itself is aimed towards blueprint developers, however you can further this in c++ if you wish and do http calls from within c++, the reason i did not add this functionality is again to help blueprint developers being able to expand on this system from entirly in blueprints where they can use plugins like va-rest etc.. to do the http calls.

Getting Started

The first thing you want to do if create a component and to do this we are going to open unreal engine and select tools then click new c++ class

After selecting new c++ class you will be presented with a option to select a c++ class, we are going to use actor as the base class for this so scroll down and select actor component and click next

after selecting this component we will be asked to give it a name, i named mine ServerComponent as you can see this already exists for me so ignore this error message.

after you have clicked next it should open visual studio for you to enter you code into the header/cpp files

FString UServerComponent::ServerPort()
{
	FString ServerPort = "Cant Get Server Port...";
	if (GetWorld()->IsServer())
	{
		ServerPort = FString::FromInt(GetWorld()->URL.Port);
	}

	return FString(ServerPort);
}

FString UServerComponent::GetPlayedMap()
{
	FString Map = GetWorld()->GetMapName();
	return FString(Map);
}

FString UServerComponent::TotalPlayers()
{
	int32 NumPlayers = 0;
	for (FConstPlayerControllerIterator Iterator = GetWorld()->GetPlayerControllerIterator(); Iterator; ++Iterator)
	{
		NumPlayers++;
	}
	return FString(FString::Printf(TEXT("%d"), NumPlayers));
}

now that we have added the component we can add the blueprint code the following uses va-rest so make sure to have this installed to your engine or change this to the http request module that you are using
